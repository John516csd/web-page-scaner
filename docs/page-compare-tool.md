# Page Diff Tool — 技术方案报告

> 网页对比 Web App：输入两个 URL，在线查看页面差异

---

## 1. 工具定位

一个**独立的 Web 应用**，打开浏览器就能用。

- **输入**：在页面上填两个 URL
- **操作**：点一下「开始对比」
- **输出**：在页面上直接看差异结果（HTTP、SEO、视觉截图对比）

### 用户体验

```
打开 http://localhost:3000
  ↓
填入 URL A 和 URL B
  ↓
点击「开始对比」
  ↓
实时显示进度（正在检查 HTTP... 正在截图...）
  ↓
页面上直接展示完整对比报告
  ↓
可选：下载 JSON / 导出 HTML
```

### 使用场景

- 网站迁移前后对比
- 新版本上线前与线上对比
- 多环境一致性检查（staging vs production）
- 非技术人员也能用，不需要命令行

---

## 2. 两种模式

工具有两种对比模式，通过 Tab 切换：

### 模式一：单页对比

对比**一对 URL**，看这两个页面之间的差异。

- 输入：URL A、URL B（两个完整的 URL）
- 输出：这一对页面在 HTTP / SEO / 视觉上的全部差异
- 场景：快速检查某个具体页面迁移后是否一致

### 模式二：站点对比

对比**两个站点**，自动发现所有页面并逐页对比。

- 输入：站点 A 的 base URL、站点 B 的 base URL
- 页面来源：
  - 从 sitemap.xml 自动抓取（输入 sitemap URL）
  - 手动粘贴路径列表（一行一个路径）
  - 两者混合
- 输出：所有页面的对比汇总 + 每个页面可展开看详情
- 场景：网站迁移后全站对比，新版本上线前全站回归

#### 分批控制

站点对比可能涉及几百甚至几千个页面，一次性全跑不现实。提供分批控制：

| 控制项 | 说明 | 默认值 |
|--------|------|--------|
| **每批页面数** | 一次对比多少个页面 | 10 |
| **自动继续** | 当前批次完成后是否自动开始下一批 | 关闭（手动点「下一批」） |
| **从第 N 个开始** | 可以指定从列表中第几个页面开始（断点续跑） | 1 |
| **优先失败项** | 重新对比时，优先跑上次失败的页面 | 开启 |

交互流程：

```
1. 输入两个站点 URL + 页面来源
2. 工具先加载页面列表，显示「共发现 156 个页面」
3. 用户设置「每批 10 个」
4. 点击「开始对比」→ 开始跑前 10 个
5. 前 10 个跑完 → 显示结果 + 按钮「继续下一批 (11-20)」
6. 用户检查结果后，点击继续 → 跑下一批
7. 也可以开「自动继续」一路跑完
8. 随时可以暂停
```

### 2.1 对比维度

| 维度 | 检查项 |
|------|--------|
| **HTTP** | 状态码、Content-Type、TTFB |
| **SEO** | title、description、canonical、og:*、twitter:*、hreflang、h1、JSON-LD、robots |
| **视觉** | Desktop (1440×900) + Mobile (375×812) 全页截图像素 diff |

### 2.2 高级选项（可折叠面板）

| 选项 | 说明 | 默认值 |
|------|------|--------|
| 视口尺寸 | Desktop / Mobile 勾选 | 两者都选 |
| 禁用动画 | 截图前注入 CSS 禁用动画 | 开启 |
| 隐藏元素 | CSS 选择器列表，截图前隐藏 | 空 |
| 等待时间 | 截图前额外等待 (ms) | 0 |
| 差异阈值 | 超过多少 % 标记为 FAIL | 15% |

## 3. 技术架构

### 3.1 前后端分离

```
┌──────────────────────────────┐
│         前端 (React)          │
│  - 输入表单                    │
│  - 实时进度展示                │
│  - 对比结果可视化              │
│  - 截图左右对比 + 差异图       │
│  - 导出 JSON / HTML           │
└──────────────┬───────────────┘
               │ HTTP API + WebSocket (实时进度)
               │
┌──────────────▼───────────────┐
│        后端 (Node.js)         │
│  - 接收对比请求               │
│  - HTTP 检查 (fetch)          │
│  - SEO 检查 (cheerio)         │
│  - 视觉截图 (Playwright)      │
│  - 像素对比 (pixelmatch)      │
│  - 返回结果 + 截图 base64     │
└──────────────────────────────┘
```

**为什么需要后端？**

Playwright（浏览器截图）只能在 Node.js 服务端运行，前端浏览器里做不了。HTTP 和 SEO 检查也存在跨域问题，需要服务端代理请求。

### 3.3 API 设计

#### POST /api/diff — 单页对比

```json
// 请求
{
  "urlA": "https://www.notta.ai/en/pricing",
  "urlB": "http://localhost:3000/en/pricing",
  "checks": ["http", "seo", "visual"],
  "options": {
    "viewports": ["desktop", "mobile"],
    "disableAnimations": true,
    "hideSelectors": [".timestamp"],
    "waitTime": 0,
    "failThreshold": 0.15
  }
}

// 响应
{
  "taskId": "abc123"
}
```

#### POST /api/site-diff — 站点对比

```json
// 请求
{
  "baseUrlA": "https://www.notta.ai",
  "baseUrlB": "http://localhost:3000",
  "paths": ["/en/pricing", "/en/blog", "/en/about", "...共 156 个"],
  "checks": ["http", "seo", "visual"],
  "batchSize": 10,       // 每批对比多少个页面
  "startIndex": 0,       // 从第几个页面开始（断点续跑）
  "options": { ... }
}

// 响应
{
  "taskId": "site-abc123",
  "totalPages": 156,
  "currentBatch": { "start": 0, "end": 10 }
}
```

#### POST /api/sitemap — 解析 sitemap 获取页面列表

```json
// 请求
{ "sitemapUrl": "https://www.notta.ai/sitemap.xml" }

// 响应
{
  "paths": ["/", "/en/pricing", "/en/blog", "..."],
  "total": 156
}
```

用途：站点对比模式下，用户填入 sitemap URL 后，先调这个接口拿到页面列表，展示给用户确认，再开始对比。

#### WebSocket /ws/:taskId — 实时进度

```json
// 单页对比的推送流
{ "type": "progress", "step": "http", "status": "running" }
{ "type": "progress", "step": "http", "status": "done", "data": { ... } }
{ "type": "progress", "step": "seo", "status": "running" }
{ "type": "progress", "step": "seo", "status": "done", "data": { ... } }
{ "type": "progress", "step": "visual", "status": "running", "message": "正在截图 Desktop..." }
{ "type": "progress", "step": "visual", "status": "done", "data": { ... } }
{ "type": "complete", "result": { /* 完整结果 */ } }

// 站点对比额外的推送
{ "type": "batch_progress", "current": 3, "total": 10, "path": "/en/about", "status": "done" }
{ "type": "batch_complete", "batchIndex": 0, "results": [...], "hasMore": true, "nextStart": 10 }
```

### 3.4 数据流

**单页对比**：

```
填入 URL A、URL B
    │
    ├── POST /api/diff ──▶ 返回 taskId
    ├── 连接 WebSocket ──▶ ws://:taskId
    │                    │
    │                    ├── HTTP 检查 → 推送进度 ──▶ 显示 ✅ HTTP 完成
    │                    ├── SEO 检查  → 推送进度 ──▶ 显示 ✅ SEO 完成
    │                    ├── 视觉截图  → 推送进度 ──▶ 显示截图三栏对比
    │                    └── complete  ────────────▶ 显示完整报告
    │
    └── 导出 JSON / HTML
```

**站点对比**：

```
填入站点 A、站点 B
    │
    ├── 输入 sitemap URL
    │     └── POST /api/sitemap ──▶ 返回 156 个路径
    │
    ├── 确认页面列表，设置「每批 10 个」
    │
    ├── 点击「开始对比」
    │     └── POST /api/site-diff ──▶ 返回 taskId (batch 1-10)
    │           └── WebSocket 推送:
    │                 ├── 第 1 个页面完成 ──▶ 表格更新一行
    │                 ├── 第 2 个页面完成 ──▶ 表格更新一行
    │                 ├── ...
    │                 └── batch_complete ──▶ 显示「继续下一批」按钮
    │
    ├── 点击「继续下一批」
    │     └── POST /api/site-diff (startIndex: 10) ──▶ 跑 11-20
    │           └── 同上...
    │
    ├── 全部跑完 ──▶ 显示汇总报告
    │
    └── [ 重跑失败项 ] [ 导出 JSON ] [ 导出 HTML ]
```

## 4. 技术选型

### 4.1 前端

| 用途 | 选择 | 理由 |
|------|------|------|
| **框架** | React + Vite | 轻量快速，开发体验好 |
| **UI 库** | Tailwind CSS | 原子化样式，不需要引入重 UI 库 |
| **截图查看** | 原生 CSS Grid | 三栏布局（A / diff / B），无需第三方图片查看器 |
| **WebSocket** | 原生 WebSocket API | 浏览器原生支持，不需要 socket.io |
| **状态管理** | React useState/useReducer | 状态简单，不需要 Redux |

### 4.2 后端

| 用途 | 选择 | 理由 |
|------|------|------|
| **HTTP 服务** | Fastify | 比 Express 更快，原生 TypeScript 支持好 |
| **WebSocket** | ws (via @fastify/websocket) | 轻量稳定的 WebSocket 实现 |
| **浏览器自动化** | Playwright | 截图稳定，支持网络空闲检测、字体等待、全页截图 |
| **像素对比** | pixelmatch + pngjs | 零依赖、速度快，GitHub 标准截图 diff 方案 |
| **HTML 解析** | cheerio | 轻量 jQuery 式 API，提取 meta/title/h1 |
| **XML 解析** | fast-xml-parser | 解析 sitemap.xml，轻量零依赖 |

### 4.4 为什么不用现成工具

| 工具 | 能力 | 缺什么 |
|------|------|--------|
| BackstopJS | 视觉回归 | CLI only；不做 SEO/HTTP；无 Web UI |
| Percy / Chromatic | 云端视觉 | 付费 SaaS；不做 SEO；不支持任意 URL 对比 |
| Lighthouse CI | 性能审计 | 单站点审计，不做双站点对比 |
| Screaming Frog | SEO 爬虫 | 桌面应用；不做视觉对比 |
| htmldiff / diff2html | HTML diff | 文本级 diff，不做视觉/SEO/HTTP 检查 |

没有现成工具能在 Web 界面上完成「输入两个 URL → 看到 HTTP + SEO + 视觉全维度差异」。

## 6. 边界情况处理

| 问题 | 方案 |
|------|------|
| 页面有动态时间戳 | 高级选项中填「隐藏元素选择器」，截图前隐藏 |
| 随机推荐 / A/B 测试 | 同上，隐藏对应区域 |
| CSS 动画 | 默认开启「禁用动画」选项 |
| 字体加载慢 | 后端截图前自动等待 `document.fonts.ready` |
| 图片懒加载 | 截图前自动滚动到底部触发所有懒加载 |
| 两个页面高度不同 | 取较大高度，短的用空白填充后再 diff |
| 页面加载慢 | 高级选项中设置「额外等待时间」 |
| 跨域 / CORS | 所有请求通过后端代理，不存在跨域问题 |
| localhost / 内网 URL | 后端本地运行，可以直接访问 localhost 和内网 |

## 10. 总结

### 一句话

**打开浏览器，填两个 URL，点一下，看差异。**

### 两种模式

- **单页对比** — 输入两个 URL，看这一对页面的差异
- **站点对比** — 输入两个站点，分批逐页对比，看全站差异

### 核心特点

- **零门槛** — Web 界面，不需要命令行，非技术人员也能用
- **两种模式** — 单页快速检查 + 站点全量对比，各取所需
- **分批控制** — 站点对比可设每批页面数，按需暂停/继续，不怕页面太多
- **实时反馈** — WebSocket 推送进度，不用干等
- **三个维度** — HTTP + SEO + 视觉截图，全覆盖
- **直观展示** — 截图三栏对比，差异一目了然
- **可导出** — JSON / HTML 报告，可分享给团队

### 技术栈

```
前端
  React       ─── UI 框架
  Vite        ─── 构建工具
  Tailwind    ─── 样式
  WebSocket   ─── 实时进度

后端
  Fastify     ─── HTTP 服务
  Playwright  ─── 浏览器截图
  pixelmatch  ─── 像素对比
  cheerio     ─── HTML/SEO 解析
  ws          ─── WebSocket
```
